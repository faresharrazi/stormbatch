Build a simple MVP web app called “StormBatch”.

Goal:
Create a small internal tool that takes:
1. a Livestorm API key,
2. an Excel file (.xlsx),
3. one or more Livestorm session IDs,

and bulk-registers all rows from the Excel file into the selected Livestorm session(s) using Livestorm’s bulk registration endpoint.

Important constraints:
- Keep it simple and working.
- No database.
- No queue/job system.
- No authentication system.
- No fancy tooling.
- No background workers.
- No Redis.
- No Celery.
- No Docker unless absolutely necessary.
- No TypeScript unless needed; plain JavaScript is fine.
- Focus on clean, readable code and a straightforward UX.

Tech stack:
- Frontend: Vue 3 + Vite
- Backend: FastAPI (Python)
- Excel parsing: pandas + openpyxl
- HTTP requests to Livestorm: httpx or requests
- Run locally with simple commands

Product behavior:
The app should:
- Let the user paste a Livestorm API key
- Let the user upload an Excel file
- Detect the Excel columns automatically
- Preview the first few rows
- Let the user map Excel columns to Livestorm people attribute IDs
- Remind the user that the Livestorm registration settings must already contain matching people attributes with the exact IDs they map to
- Let the user paste one or multiple session IDs
- Submit a bulk registration job for each session ID
- Poll the Livestorm job endpoint until the job finishes
- Fetch and display final task results
- Show success/failure status in a simple way

Important Livestorm API notes:
- Use the Livestorm bulk registration endpoint: POST /v1/sessions/{id}/people/bulk, which creates an asynchronous job rather than returning the final registration result immediately. The official docs describe it as a job-based mechanism and document a 202 Accepted response. :contentReference[oaicite:0]{index=0}
- After creating the job, poll the related job endpoint and then fetch the job tasks/results when the job is finished, following the job flow described in Livestorm’s documentation/changelog. :contentReference[oaicite:1]{index=1}

Backend API design:
Implement these FastAPI endpoints:

1. POST /api/preview
Purpose:
- receive uploaded Excel file
- read the first sheet
- return:
  - detected headers
  - preview rows (first 5 rows)
  - normalized header suggestions

2. POST /api/register
Purpose:
- receive:
  - api_key
  - session_ids
  - mapping
  - parsed rows or re-read file
- validate everything
- build Livestorm payloads
- create one Livestorm bulk job per session ID
- return the created job IDs

3. POST /api/job-status
Purpose:
- receive:
  - api_key
  - session_id
  - job_id
- fetch current Livestorm job status
- if finished, optionally also fetch tasks
- return a simplified object for the UI

Livestorm payload building:
Transform each row into this structure:

{
  "data": {
    "type": "jobs",
    "attributes": {
      "tasks": [
        {
          "fields": [
            {
              "id": "email",
              "value": "john@example.com"
            },
            {
              "id": "first_name",
              "value": "John"
            },
            {
              "id": "company",
              "value": "Acme"
            }
          ]
        }
      ]
    }
  }
}

Rules for payload generation:
- Build one task per Excel row
- Build one bulk job request per session ID
- Skip ignored columns
- Skip empty values for optional fields
- Do not skip email if it is empty; fail validation before submission
- Preserve string values cleanly
- Convert NaN/null to empty or skip appropriately

Polling behavior:
- The frontend may poll the backend every 2–3 seconds after job creation
- The backend should call Livestorm job endpoints and return:
  - pending / processing / ended / failed if applicable
- Once ended, fetch job tasks and show per-row outcomes if available

Error handling:

- Examples:

  - “Excel file is empty”
  - “Livestorm API request failed”
  - “One or more mapped attribute IDs may not exist in Livestorm registration settings”
- Backend should return readable JSON errors
- Frontend should show them without crashing
